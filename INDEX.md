> **「INDEX.md を読んで」** と言われたら、下記フローに従って自律的に開発を進めてください。

---

## ⚡ トリガー：「INDEX.md を読んで」で自動開発開始

このコマンドを受け取ったら、AI は次の流れを自動実行します。

1. **プロジェクト分析** — `README.md` / `docs/` / コード / GitHub Issue / `data/` の状況を把握
2. **開発計画立案** — `docs/roadmap.md` の未完了タスクを優先度順に整理し、次に着手すべき Issue を1つ提案
3. **承認待ち** — ここで停止し、ユーザーの「承認します」を待つ
4. **自動実行** — 承認後：実装 → 自己チェック（lint/動作確認）→ コミット用の差分提示 → 次タスクへ

> このプロジェクトの開発者はスマホ（GitHubアプリ）からコミットします。
> AI はコードを生成し、ユーザーが承認・コミットする「半自律ループ」で進めます。

---

## 🎯 このプロジェクトのゴール（絶対に外さない前提）

- **競馬の分析ツール**。馬券の**購入は手動**（ユーザーが自分でやる）。**購入の自動化は実装しない・対象外**。
- **個人利用**。商用化・収益化はしない。
- **原則無料**で完結（GitHub / GitHub Actions / Vercel の無料枠、データは netkeiba 収集）。
- **スマホだけ**で開発も使用もできる（フロントはモバイル最適化、運用はクラウド自動実行）。
- **分析内容・結果・根拠が「よくわかる」設計**。数字の羅列でなく「なぜそう判断したか」まで見せる。

詳細は `docs/requirements.md` を参照。

## 🔄 自動化の対象とAIの権限

| やってよい ✅ | やってはいけない ❌ |
|---|---|
| コード実装・修正・リファクタ | 馬券購入・IPAT連携・投票自動化の実装 |
| 分析ロジックの作成・改善 | 生の netkeiba データを `data/` にコミット（後述） |
| ドキュメント更新 | 有料サービス前提の実装（JRA-VAN等） |
| GitHub Issue の作成・整理 | 個人情報・認証情報のハードコード |
| Actions ワークフローの修正 | スコープ外の大規模変更を無断で進める |

## 🧭 優先度の判断基準

```
critical : ビルド/Actions失敗、データ収集が完全に止まる、セキュリティ問題
high     : 収集や分析の不具合、表示が壊れる、誤った予想が出る
medium   : 新機能（新しい分析軸・新しい券種対応・UI改善）
low      : ドキュメント整備、軽微なリファクタ、コメント追加
```

## 🔧 リポジトリ情報・実行設定

- **Repository**: `<your-username>/kachiuma`（作成後にここを書き換える）
- **Main Branch**: `main`
- **データ方針（重要）**:
  - 公開リポジトリのため、**生の netkeiba データ（出馬表・結果・過去走の生テーブル）は `data/raw/` に置き、`.gitignore` で除外**。Actions 内の一時成果物としてのみ扱う。
  - **コミットしてよいのは「自作の分析出力」**（`data/predictions/*.json`＝印・期待値・スコア・根拠）と**サンプルデータ**のみ。
- **分析対象ファイル**: `README.md`, `docs/`, `scraper/`, `analysis/`, `web/`, `.github/workflows/`

### 開発・確認コマンド

```bash
# データ収集（ローカル確認用。本番はActionsが自動実行）
python scraper/collect.py --date 20260621

# 分析（予想＋期待値の算出）
python analysis/predict.py --in data/raw --out data/predictions

# フロント確認（ローカルサーバ）
cd web && python -m http.server 8000
```

## 📝 ループのお作法（INDEX.md の保ち方）

1. タスクを1つ完了したら、`docs/roadmap.md` のチェックを更新し、上記「現在地」も最新化する。
2. 新しく判明した課題は Issue 化（または roadmap に追記）してから着手する。
3. 仕様の判断に迷ったら、勝手に進めず**ユーザーに1問だけ確認**する。
4. 1回のループで触る範囲は1 Issue 分に絞る（巨大PRを避ける）。
